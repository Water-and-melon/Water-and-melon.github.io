cms收集器

发表于 2017-12-31 | 分类于 java

前言

CMS(Concurrent Mark Sweep),应用广泛。侧重点是，希望系统的停顿尽可能的少，特别对于一些需要实时响应的场景下较为适合。

流程

CMS收集器的运作过程主要有以下几步（结合图）：

  • 初始标记(CMS initial mark)
  • 并发标记(CMS concurrent mark)
  • 重新标记(CMS remark)
  • 并发清理(CMS concurrent sweep)

下面通过一个打扫房间的实际例子解释下CMS。

好比房间里有5个熊孩子在玩耍，他们会把房间搞的很乱，你在一边要实时的打扫房间。你肯定不能把熊孩子赶出房间再打扫卫生，这时，你可以这样做：

  • 首先，你会让熊孩子都停一下，这时候你迅速的清理一些垃圾。孩子暂停打闹的时间是很短的，所以这里的关键是找到一些明显的，一下子就看到的垃圾一起扫到一边。这一步就对应初始标记。
  • 处理完一些明显的垃圾，你就要处理比较细微的垃圾，此时，你不需要孩子停下来，你自己在一边慢慢的把垃圾扫到一起。这一步对应并发标记。
  • 眼看就要大功告成了，这时你发现在你处理细小垃圾的时候，熊孩子又制造出了一些垃圾，好在这些垃圾数量不多。这时，你就会大喊一声：stop！让熊孩子再停一下，然后快速把新产生的垃圾扫到一起。这一步对应重新标记。
  • 总算大功告成了，你就把扫到一起的垃圾装入垃圾袋，完成打扫。这一步对应并发清理。

优缺点

优点：并发收集，低停顿。(你在扫垃圾的时候，熊孩子还是可以玩耍的)

缺点：

1.CMS长期占用一部分的线程，导致应用的吞吐率降低。（你在一边打扫的时候，熊孩子感觉玩的不爽）

2.CMS收集器无法处理浮动垃圾。用于CMS并发清理的时候，用户还在执行，垃圾一直清理不干净。有些时候会触发另一次的fullGC。（你发现有熊孩子在，怎么都收拾不干净，只好把熊孩子轰出去再打扫）

3.CMS收集器是一款基于“标记–清除”实现的算法，会产生很多碎片。改进的方法就是进行合并整理，这样就会导致停顿时间变长。

总结

CMS不太完美。所以发明了另一种G1收集器。

分布式系统原理(1)

发表于 2017-12-28 | 分类于 分布式原理

前言

在分布式的环境下，数据分散在不同的节点上。数据分散的方式有多种，本文做一个总结。

hash方式

hash方式就类似于hash表，每台机器就是一个哈希表中的桶，根据hash值分布到各个桶中。这种方式的关键点在于：

  • hash函数的选择。如果hash函数选择的好，数据均衡落在节点上，这就非常完美。但是，考虑极端的情况下，hash函数将所有的数据都落在一个节点上，那就糟糕透了。

hash方式的好处很明显：

  • 实现简单
  • 元数据的管理很容易，根据hash函数就可以计算数据所在节点。

hash方式的缺点也很明显：

  • 如果集群好扩容，就要大量的移动数据，牵一发而动全身，这样的代价是很高的。为了优化这一点，扩容的时候一般选择成倍的增加节点，以减少数据移动。（java中的hashmap也采用了这种做法）
  • 很容易造成数据倾斜。假设存储用户数据，根据用户id进行hash，若一个用户的数据量特别大时候，就会造成倾斜。为了缓解数据倾斜，一般的做法是多增加几个维度。例如用户数据中，可以根据id+数据记录时间进行hash，也可以在id后面随机添加n位数字。但这游破坏了数据的关联性，相同id的数据被分到了多个节点上。

按数据范围

这种方式根据数据中的可排序的字段进行范围划分，以时间为例：第一季度的数据存在第一个节点，第二季度的数据存在第二个节点，以此类推。个人感觉这种方式和hash的做法较为类似，但也有很多不同：

  • hsah的分法你只需要记录hash函数即可，按范围则要记录范围和节点一一对应的关系，维护元数据的开销要大很多。
  • 相比于hash方式，按数据范围划分的方式更加灵活。范围的区间大小是可以调整的，拿上面的例子。如果第一季度的数据量特别大，我可以对这个范围进一步细化，变为第1月，第2月，第3月。区间可以根据数据量动态调节。
  • 扩容也更加简单。若第二季度的数据量快速增长，可以为这个区间增加一个节点，这时，区间范围就变成了第4、5月，第6月，只需把原来节点的第6月数据移动到新增节点就完成了扩容。

一致性hash

个人认为：一致性hash是按数据范围和按hash方式的结合体，非常巧妙。
有些时候，对于一些数据你很难根据某些字段进行排序，考虑根据hash值进行范围划分，这样一个节点上存储着是一个hash范围的数据。把范围用一个环代替，节点是落在环上的点。如下图所示：

对数据进行hash，找到环上对应的点，该数据就存储在顺时针方向的第一个节点上。这种做法就结合了按数据范围分布的优势，动态调节范围、扩容等操作都很方便。但是在负载均衡上还不是特别完美。为此又进行了改进，引入虚拟节点。

其原理是，一个节点可以对应多个虚拟节点，虚拟节点在环山是随机分布的。数据存储时，先找到顺时针方向的第一个虚拟节点，在存储到虚拟节点对应的存储节点中。这种做法的出发点是：

  • 范围的力度精细，能更好的负载均衡。
  • 对于数据堆积在一个范围的情况下，虚拟节点也可以把这些节点分散到不同的存储节点上。相当于多了一次打散数据的机会。

按照数据量分布

考虑到像大文本的这种数据，他的特点是单个数据量大，用之前的方式都很容易造成负载不均。这时，我们会考虑将数据切成若干个数据块，再结合之前的三种方在对这些数据块进行存储。这种方式元数据的管理较为复杂。

实际案例对照

  • hdfs：按照数据量，进行切分（通常128M）。具体放置在哪个节点通常根据namenode进行决策。
  • kafka：kafka针对一个partition选择Broker采取的是随机的方式（和hash方式类似）。
  • Hbase：按数据范围分布。抛开hdfs这个底层层面，Hbase会对rowkey进行排序，一个范围内的rowkey是存储在一起的。
  • redis：一致性hash。

基本上都能对应上，细节上都有各自的优化。

网络安全——加密+认证

发表于 2017-12-15 | 分类于 计算机基础

今天了解了一下网络安全方面的内容，就加密和认证做一个总结。

无加密

一开始人们直接通过明文进行通讯，导致大量的信息窃听，人们认识到这个严重的问题，才有了加密。

对称加密

对称加密的想法是，发送方把明文通过一定的规则的转换，转换成一堆看上去毫无意义的密文。接收方通过规则把密文反向的转换成发送方的明文，完成通讯。这样，就算中间有人窃听，他也听不懂。但这存在一个问题，如何让通讯双方拥有相同秘钥呢（转换规则）？

非对称加密

显然，在复杂的网络环境下，通讯双方协商出相同的秘钥存在困难。这时，就有研究者提出了新的解决方案。接受信息的一方通过算法生成两个秘钥，一个是公有的，可以告诉互联网的所有人，一个是私有的，只有自己知道。发送方通过公钥对内容进行加密，接受方通过私钥解密可以恢复出原来的内容。这个方法很好的解决了对称加密协商秘钥的困难。然而，非对称加密也存在着问题。

  • 非对称解密的需要消耗大量的计算资源，速度慢。
  • 公钥所有人都可以拿到，坏人可以轻易的给接受者发送消息，而且接受方也不能判断出发送者是坏人。

对称加密+密钥交换协议

由于非对称加密存在上面的问题，人们还是更希望使用对称加密的方法。这时，就有人提出了一种协商密钥的方法（DH算法），通讯双方可以协商出一个秘钥，而且秘钥不会被他们知道。这也就基本上解决了对称加密遗留下来的问题。但是，这还是有个问题。假设存在下面的情形：一个攻击者在通讯双方协商秘钥的时候，参杂到通讯中间，分别拦截协商秘钥的信息，并且假装是协商的另一方协商秘钥。此时，攻击者就可以分别和发送方和接收方通过秘钥进行通讯，为所欲为，更重要的问题是通讯双方完全不知情。这就是著名的中间人攻击。

签名认证+密钥交换

前面这个问题关键缺陷在于，不能分辨出中间夹杂的那个坏人。所以问题关键在于如何认证通讯的对方是个好人。
这时，人们就想出了一个非常精妙的办法：

  • 接收方根据之前提到的非对称加密的方法，生成公钥+私钥。其中，为了验证自己的身份，接收方会用自己的一些个人信息生成一个签名，然后用生成的私钥进行加密，然后把签名+加密后的签名交给发送方。发送方根据公钥，进行解密，与开始的签名进行对照，如果相同，就说明对方是好人。然后就可以进行下一步的秘钥交换。然后双方就用秘钥，用对称加密的方式进行通讯。

但是，这还有一个问题。就是坏人也可以这样干啊，给自己打一个好人的标签。这个时候，唯一的办法是，找一个权威的机构（CA），如果判定是好人，就给他打个标签，用机构的私钥进行加密，并把公钥发给发送方。进一步考虑，公钥在传输的过程中可能破坏，所以只能一开始就让发送客户端知道。这时，只能依靠一些知名的客户端操作系统，他们会把CA机构的公钥事先植入，这样就可以保证公钥的安全。
