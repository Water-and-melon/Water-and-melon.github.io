hadoop shuffle 和 spark shuffle 的区别

发表于 2017-10-21 | 分类于 大数据

前言

在批处理的大环境下，spark和hadoop的运用都很广泛，spark比起hadoop具有更好的性能。其中shuffle性能的好坏直接影响了整个处理环节的好坏，本文就对hadoop shuffle 和 spark shuffle 做一个比较。

hadoop shuffle

  1. IFile存储格式：(key-len , value-len , key ,value)
  2. map端的排序：会存在一个缓冲区，当缓冲区利用的比列到达一定的阈值后，会进行一次排序。然后写入到磁盘中，这样就存在很多的有序的小文件，最后，对这些文件进行一次归并排序，形成一个有序的大文件。（缓冲区的排序使用的是快速排序，有一定的优化：枢轴的选取，递归次数的控制），文件的归并使用的是堆排序）
  3. spill的具体过程： 首先对缓冲池中的数据进行快速排序，快速排序的依据（首先针对parition，然后根据key进行排序），这样就保证每个partition的内容都是有序的。当写入到磁盘的时候，如果map有设置了combiner，写入磁盘前先做一次聚合。然后写入到磁盘，当然，也要记录该文件针对分区的索引文件。
  4. Combine需要把所有的小文件进行一次合并。合并成一个大文件和一个索引文件，一个文件的好处是可以有效的避免随机IO开销。
  5. shuffle阶段：抓取对应partition的数据，如果超过设定的内存大小，写到磁盘。
  6. merge：在远程拷贝数据的同时，进行合并，减少内存和磁盘文件的数量。
  7. sort：因为map端已经排序过了，所以只需要简单的归并即可。
  8. reduce：处理自定义逻辑。

副本一致性

发表于 2017-07-16 | 分类于 分布式原理

前言

分布式系统中，为了解决数据丢失的问题，只能采取多副本进行冗余。副本包括数据副本和服务副本。副本一致性简单的理解就是，分布式系统整个集群在某种约束条件下，对外提供的副本是相同的。根据一致性的强弱可以大致分为一下几种一致性级别。

强一致性

任何时刻任何用户或节点都可以读到最近一次成功更新的副本数据。强一致性是程度最高的一致性要求。
可以结合一下hdfs的设计，hdfs是多副本的，个人认为是强一致性的。写文件的时候会去申请一个租约，这保证了针对一个文件，只能有一个client去写，但是一个client在写的过程中，涉及到多个副本间的同步，而且如果是append操作，该文件对读也是可见的，为了做到强一致性，hdfs的做法就是全部append完了，append内容才对外可见。
paxos协议也是属于强一致性的范畴。

弱一致性

弱一致性的一致性级别最低，就是在读数据的时候，多个副本数据不一致，且不管在哪个副本上都不一定能读到最新的更新值。

poll,select和epoll

发表于 2017-05-16 | 分类于 Linux

前言

IO复用使得程序能够同时监听多个文件描述符，当存在大量连接的时候能够提升系统性能。Linux下实现IO复用的系统调用有select，poll，epoll三种。本文对这三种进行总结。

select

首先select阻塞的（一开始我一直以为是非阻塞）。select就是在一定时间内，对用户设置的可读，可写等事件进行监听。

  • nfds：指的是监听的描述符的个数，有限制，32位机器是1024，64位机器是2048。
  • readfds writefds errorfds分别对应自己关心事件对应的描述符，如readfds里就是关心是否可读的描述符的集合。select也是通过修改集合来通知程序哪个文件描述符已经就位。
  • timeout就是select的超时时间。select将关心的文件描述符拷贝到内核态地址，并通过轮询的方式，查看集合中的文件描述符是否有对应的事件发生，如果有发生，就把对应的位置进行标记，以通知应用程序，select最后返回的是以就绪的线程描述符的数量。如果遍历一遍没有就绪的线程，内核就会一直的轮询，直到有对应的事件发生。如果一直没有，且超过了超树事件，就返回0。
